## 2. GitHub 上のフォルダ構造

GitHub リポジトリ `BST1120/yusha` の中身は、次のような階層です。

```text
yusha/                          ← リポジトリのルート（ここがWebサイトのトップ）
│
├─ index.html                   ← トップページ
├─ lyrics.html                  ← 歌詞一覧
├─ lyrics-track.html            ← 曲ごとの歌詞（1枚で全曲対応）
├─ archive.html                 ← 制作記録トップ
├─ archive-track.html           ← エピローグ記事ページ（4項目のみ）
├─ download.html                ← ダウンロードページ
│
├─ css/
│  └─ style.css                 ← 見た目・レスポンシブ（触る必要はほぼない）
│
├─ js/
│  ├─ config.js                 ← ★ ダウンロードURLはここ
│  └─ main.js                   ← ページの動き（触る必要はほぼない）
│
├─ data/
│  └─ tracks.json               ← 曲名・曲順・エピローグ項目の一覧
│
├─ assets/
│  ├─ images/                   ← ★ ジャケット・ステッカー等の画像
│  ├─ text/                     ← ★ 歌詞・制作記録・エピローグの本文
│  │  ├─ lyrics/                ← 歌詞（曲ごと .txt）
│  │  ├─ production/            ← 曲別（Web未使用・退避用）
│  │  ├─ epilogue/              ← エピローグ（4項目 .txt）
│  │  └─ remaster-note.txt      ← リマスターノート
│  ├─ audio/                    ← 試聴用（現在は未使用）
│  └─ video/                    ← 背景動画（任意）
│
├─ scripts/
│  └─ publish.ps1               ← ★ GitHub へ送るボタン代わり
│
└─ site.config.json             ← 公開URL・リポジトリ名の設定
```

**重要:** `index.html` はリポジトリの**いちばん上**にあります。  
サブフォルダの中にだけ置くと、サイトが 404 になります。

## 3. どうやって動いているか（非エンジニア向け）

### 3-1. 3種類のファイル

| 種類 | 役割 | 例 |
|------|------|-----|
| **HTML** | ページの骨組み・リンク | `index.html`, `lyrics.html` |
| **CSS** | 色・字体・レイアウト・スマホ対応 | `css/style.css` |
| **JavaScript (JS)** | 歌詞ファイルの読み込み、一覧の生成 | `js/main.js` |

**データベースはありません。** 歌詞などは `.txt` ファイルに書いておき、JS が読み込んで表示します。

### 3-2. ページのつながり（イメージ）

```text
index.html（トップ）
    ├→ lyrics.html → lyrics-track.html?slug=2f  （2F の歌詞）
    ├→ archive.html（年表のみ）→ archive-track.html?slug=ep-sound  （エピローグ）
    └→ download.html  （外部リンクへ飛ぶ）
```

### 3-3. GitHub Pages の役割

1. あなたがファイルを GitHub に **push** する  
2. GitHub が `main` ブランチのファイルを **Webサーバーとして公開** する  
3. URL `https://BST1120.github.io/yusha/` で誰でもアクセスできる  

**ホスティング料は無料**（Public リポジトリの場合）。  
ファイルが GitHub 上に残る限り、URL は基本的に使い続けられます。

## 4. スマホ対応（レスポンシブ）の仕組み

**はい、HTML と CSS で対応しています。** 専用のスマホアプリや別サイトではありません。

### やっていること

1. **HTML の `<meta name="viewport">`**  
   スマホブラウザに「PC用に縮小表示しないで」と伝える（各 HTML に設定済み）

2. **CSS の基本ルール**  
   - 画像は `max-width: 100%` で画面幅に収める  
   - 本文エリアは `max-width: 860px` で中央寄せ  
   - 文字サイズは `clamp(...)` で画面に合わせて可変  

3. **CSS の `@media`（画面幅で切り替え）**  
   - 幅 600px 以上 → ジャケット2枚を横並び  
   - 幅 480px 以下 → 余白・文字サイズを微調整  

4. **1つの HTML/CSS で PC・タブレット・スマホ共通**  
   別ファイルを作っていません。同じ `style.css` が幅に応じて見た目を変えます。

**普段の更新で CSS を触る必要はほぼありません。**

## 5. ★ 変更したいとき — ここだけ触ればOK

### 早見表

| 変えたい内容 | 触るファイル | 触らない方がいい |
|--------------|--------------|------------------|
| **歌詞** | `assets/text/lyrics/02-2f.txt` など | HTML / JS |
| **エピローグ（公開）** | `assets/text/epilogue/` または `workspace/epilogue/` | — |
| **曲別ラフ（非公開）** | `workspace/tracks/{曲}/draft/` | Web には載せない |
| **リマスターノート** | `assets/text/remaster-note.txt` | — |
| **エピローグ・音源制作** | `assets/text/epilogue/01-sound-production.txt` | — |
| **エピローグ・フィジカル製作** | `assets/text/epilogue/02-physical-production.txt` | — |
| **エピローグ・発想** | `assets/text/epilogue/03-ideation.txt` | — |
| **エピローグ・発送** | `assets/text/epilogue/04-shipping.txt` | — |
| **ダウンロードリンク（WAV等）** | `js/config.js` の `url:` | — |
| **ジャケット・ステッカー画像** | `assets/images/` 内を差し替え（同名で上書き） | — |
| **曲別の任意画像** | `assets/images/tracks/02-2f.jpg` など | — |

### 歌詞ファイル名の対応表

| ファイル | 曲 |
|----------|-----|
| `01-inner-image.txt` | inner image (intro) |
| `02-2f.txt` | 2F |
| `03-sappo.txt` | sappo |
| `04-yusha.txt` | 遊者 |
| `05-rooftop.txt` | rooftop (outro) |
| `06-orenoheya.txt` | 俺の部屋 |
| `07-tower.txt` | tower |
| `08-liquid.txt` | Liquid |

ファイル名の `{番号}-{slug}` は `data/tracks.json` と一致している必要があります。**ファイル名を変えるときは Cursor に相談**してください。

### ダウンロード URL の設定例（`js/config.js`）

```javascript
{
  id: "wav",
  label: "WAV音源データ",
  url: "https://drive.google.com/...",  // ← ここに共有リンク
},
```

`url: ""` のままだと「準備中」と表示されます。

## 6. 更新の手順（いつもこれ）

### 方法A — Cursor に頼む（いちばん簡単）

1. 歌詞やエピローグの `.txt` を編集  
2. Cursor のチャットで **「公開して」** と送る  
3. 1〜2分後、スマホで https://BST1120.github.io/yusha/ を確認  

### 方法B — 自分で push する

```powershell
cd e:\somosomo\apps\yusha-site
.\scripts\publish.ps1 -Message "歌詞を更新"
```

### ローカルで確認したいとき（公開前）

```powershell
cd e:\somosomo\apps\yusha-site
python -m http.server 8080
```

ブラウザで http://localhost:8080 を開く。  
**HTML をダブルクリックで開く（file://）だけでは歌詞が読めない**ので注意。

## 7. 永久保管するときの注意点

### 7-1. 絶対に守りたいこと

| 項目 | 理由 |
|------|------|
| **GitHub アカウント（BST1120）を削除しない** | 削除するとサイトも消える |
| **リポジトリ `yusha` を削除しない** | 同上 |
| **リポジトリを Private にしない** | 無料 Pages は Public が前提 |
| **Settings → Pages が `main` / `(root)` のまま** | 設定を変えると 404 になりうる |
| **QR・封入紙の URL を変えない** | 印刷後は変更不可 |

### 7-2. 二重のバックアップ

| 場所 | 役割 |
|------|------|
| **GitHub（BST1120/yusha）** | 公開用・本番 |
| **PC（`e:\somosomo\apps\yusha-site\`）** | 編集用・ローカルコピー |

Somosomo フォルダごとバックアップしておくと、PC が壊れても復元しやすいです。

### 7-3. 大きな音源ファイルは GitHub に置かない

- **WAV 全曲 ZIP** → Google Drive / Dropbox 等  
- サイトには **リンクだけ**（`js/config.js`）  
- GitHub に置くと URL を知った人が誰でも DL できてしまう  

### 7-4. 長期運用の心構え

- **テキスト更新（歌詞・エピローグ）** → いつでも `.txt` を編集して push  
- **デザイン変更** → `css/style.css`（Cursor に依頼推奨）  
- **GitHub 自体のサービス終了** → 可能性は低いが、その時は別ホストへファイル一式を移せばよい（静的HTMLなので移行は比較的簡単）

## 8. 触らない方がいいもの（壊れやすい）

| ファイル | 理由 |
|----------|------|
| `js/main.js` | 歌詞読み込み・一覧生成の中枢 |
| `archive-track.html` | エピローグ専用（曲 slug は無効） |
| `data/tracks.json` | 曲順・slug・エピローグ項目の定義 |
| `css/style.css` | 見た目全体（意図的なデザイン変更以外） |

**「文章だけ変えたい」なら `assets/text/` だけで十分です。**

## 9. 困ったとき

| 症状 | 確認すること |
|------|--------------|
| 404 になる | GitHub → Settings → Pages が有効か。1〜3分待ったか |
| 歌詞が「読み込めませんでした」 | ファイル名が `04-yusha.txt` 形式か。`data/tracks.json` が push されているか |
| 変更が反映されない | push したか。ブラウザを再読み込み（スマホはタブを閉じて開き直し） |
| 画像が出ない | ファイル名の大文字小文字が一致しているか |
